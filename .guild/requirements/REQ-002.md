# Deadline Mechanic

## Summary

Add an optional "Time-Sensitive Directive" mechanic to certain Prescripts.

If the timer reaches 00:00 before the user clicks Execute:
- Trigger the audio breakdown effect
- Automatically mark the task as Diverged
- Subtract 5 Proximate Trust as a direct penalty

Apply this mechanic to Medium and Hard prescripts with randomized countdown windows (for example 30 minutes or 1 hour).

Display it as a sleek, ticking countdown bar using Electronic Accent Blue (`#86bada`).

## User Stories

1. As a player, I want certain `Medium`/`Hard` prescripts to carry a visible deadline so urgency affects my decisions.
2. As a player, I want randomized deadline durations (e.g., 30m or 60m) so time pressure feels less predictable.
3. As a player, if I fail to Execute before the timer expires, I want the system to auto-mark the directive as Diverged and apply a trust penalty consistently.
4. As a player, I want a themed audio breakdown effect and visual timer treatment when a directive expires so feedback is immediate.
5. As a maintainer, I want the time-sensitive flag and timer duration represented in prescript/task state so behavior is deterministic and testable.

## Acceptance Criteria

1. Time-sensitive directives are optional and applied only to eligible `Medium`/`Hard` prescripts.
2. Timer duration is selected from approved values (initially 30 or 60 minutes) and persists for the active directive.
3. Countdown is displayed as a ticking progress bar with accent color `#86bada`.
4. If timer reaches `00:00` before Execute:
   - directive is automatically marked as Diverged
   - trust is reduced by exactly 5
   - audio breakdown effect is triggered once
5. Manual Diverge behavior remains unchanged for non-timeout cases.
6. Timeout handling is idempotent (no duplicate penalties/events if multiple effects fire).
7. Tests cover at least one timer-expiry flow and one successful Execute-before-expiry flow.

## Edge Cases

- App reload during an active timed directive should preserve remaining time or safely restart according to chosen product rule.
- Background tab throttling should not break timeout correctness (use epoch timestamps rather than animation tick counters).
- Timers should not run for `Easy` directives.
- Setting audio muted should suppress audible breakdown while keeping state transitions intact.
- Execute click at the exact expiry boundary should be deterministic (document precedence rule).

## Technical Considerations

- Extend active prescript/task model with optional `deadlineAt` and `isTimeSensitive` fields.
- Use `Date.now()`-based remaining-time calculation to avoid interval drift.
- Keep timeout penalty constant centralized with trust award constants.
- Add cleanup to prevent orphaned intervals/timeouts on directive swap/unmount.
- Keep rendering concerns isolated in prescript card/deadline component.

## Out of Scope

- Push notifications or system-level alarms.
- Server-synchronized deadlines across devices.
- Difficulty rebalance unrelated to timeout penalty.
