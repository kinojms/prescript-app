# Mobile Responsiveness

## Summary

The prescript-app renders a modal-style shell component (`HermesShell`) that frames all primary content against a full-screen cipher background (`CipherBackground`). On mobile devices in portrait orientation, this shell currently does not expand to fill the available viewport height — users see a small, under-sized panel that wastes most of the screen and makes interaction uncomfortable.

The goal is to make `HermesShell` occupy most of the visible viewport on mobile while leaving a small, tasteful margin on all four sides. This margin frames the shell against the cipher background, giving a deliberate "framed" aesthetic rather than a cramped or edge-to-edge appearance. The exact margin size is developer judgment — the intent is "make it look good," not a fixed pixel value.

A secondary issue exists in landscape orientation on mobile: the shell is currently stretched or mis-sized. It must adapt gracefully to a horizontal viewport so all content remains readable and the shell stays within viewport bounds. The minimum supported device width is 360px, covering all common modern smartphones.

## Assumptions

- **Scroll behavior:** Content that overflows the shell scrolls within the shell itself, not the page. The shell stays within the viewport; the page does not scroll. This is inferred from context — the architect should flag this if it conflicts with the existing implementation.
- **Tasteful gap:** The margin between the shell border and the viewport edge is left to developer judgment. There is no numeric target. The visual standard is: the cipher background should be clearly visible as a border/frame around the shell, but the shell should be noticeably large.
- **Header button issues** (RankHeader icons clipping or wrapping on small screens) are a secondary concern. They are in scope but lower priority than the shell sizing fix.

## User Stories

### US-1: Portrait viewport — shell fills available height

**As a** mobile user holding my phone in portrait orientation
**I want** the HermesShell to expand to nearly fill the full screen height
**So that** I have maximum reading and interaction space without wasted viewport

**Acceptance Criteria:**

1. Given a device viewport of at least 360px wide in portrait orientation,
   When the app loads or the user navigates to the main app screen,
   Then the HermesShell panel occupies the majority of the viewport height with a visible, consistent margin on all four sides (top, right, bottom, left).

2. Given a portrait viewport at 360px wide,
   When the HermesShell is rendered,
   Then the cipher background (`CipherBackground`) is visibly present around all four sides of the shell — it is not hidden behind or clipped by the shell.

3. Given a portrait viewport at 360px wide,
   When the HermesShell is rendered,
   Then the shell does not extend beyond the visible viewport (no horizontal or vertical overflow that causes the page itself to scroll).

4. Given a portrait viewport at a common larger phone size (e.g., 390px, 430px),
   When the app loads,
   Then the shell scales proportionally so the cipher background margin remains tasteful at all widths — it does not become disproportionately large or small.

**Edge Cases:**
- Very narrow device (360px): Shell still shows the cipher background margin; nothing clips off-screen.
- User resizes browser window (desktop simulation of portrait): Shell re-adjusts without requiring a page reload.

---

### US-2: Landscape orientation — shell adapts without stretching

**As a** mobile user rotating my device to landscape orientation
**I want** the HermesShell to fit within the landscape viewport without distortion or overflow
**So that** I can use the app comfortably in either orientation

**Acceptance Criteria:**

1. Given a mobile device in landscape orientation (viewport height typically 360–430px, width 640–932px),
   When the app loads or the device rotates,
   Then the HermesShell fits fully within the viewport — no content is cut off, and no scrollbar appears on the page body.

2. Given landscape orientation,
   When the HermesShell is rendered,
   Then all interactive elements (header buttons, prescript card, record panel) are reachable without horizontal scrolling.

3. Given landscape orientation,
   When the HermesShell is rendered,
   Then text and controls are readable at normal font sizes — no unintentional zooming, overflow, or squishing.

4. Given landscape orientation,
   When the app is in use,
   Then rotating back to portrait restores the portrait layout correctly without requiring a reload.

**Edge Cases:**
- Landscape at minimum height (~360px tall): Shell remains within viewport; content does not overflow below the fold.
- Rotation during an active prescript: App state is preserved across orientation change; the user does not lose progress.

---

### US-3: Header button area — controls accessible on small screens (secondary priority)

**As a** mobile user on a 360px-wide device
**I want** the RankHeader icons (mode toggle, BGM mute, settings) to remain accessible and not clip or wrap awkwardly
**So that** I can access app controls without difficulty

**Acceptance Criteria:**

1. Given a viewport of 360px wide,
   When the RankHeader is rendered,
   Then all header action buttons are visible and tappable with no clipping against the shell border.

2. Given a viewport of 360px wide,
   When a user taps a header button,
   Then the tap target is large enough to activate reliably (minimum 44×44px logical pixels per WCAG 2.5.5 AAA guidance or platform equivalent).

3. Given a viewport of 360px wide,
   When the RankHeader is rendered,
   Then rank/trust display text does not overflow its container or overlap with the action buttons.

**Edge Cases:**
- Very long rank label text: Text truncates or wraps without pushing buttons off-screen.

## Technical Considerations

- `HermesShell` currently uses `min-h-dvh` on mobile (no `md:` prefix) which forces full viewport height but does not constrain the visual frame — the border-only sides (`border-x-2`) create the appearance of a tall, unconstrained strip. The fix likely involves restructuring the outer wrapper so it centers a bounded, nearly-full-height element rather than stretching edge-to-edge vertically on mobile.
- `dvh` (dynamic viewport height) units are preferred over `vh` on mobile to account for browser chrome (address bar show/hide). The codebase already uses `dvh` — this should be preserved.
- `CipherBackground` is a sibling of `HermesShell` in `App.jsx`, rendered as a fixed/absolute layer. The gap requirement means the shell must not cover the full viewport — some visual clear space on all sides must remain uncovered.
- Landscape layout may need separate breakpoint treatment. The current `md:` breakpoints target tablet/desktop widths, not landscape-phone viewports. A `landscape` media query or orientation-aware Tailwind variant may be needed.
- The fix should not break the existing desktop layout (`md:` and above), which already uses a centered, max-width bounded shell with rounded corners, shadow, and vertical centering — that behavior is correct.
- `SettingsDrawer` is positioned outside `HermesShell` in the DOM. Any viewport-constrained layout change should verify the drawer still renders correctly over the shell.
- Scroll behavior assumption: if inner-shell scrolling is not already implemented, the `overflow-hidden` on the shell (currently applied only at `md:`) may need to be revisited — but this is an architect decision.

## Out of Scope

- Desktop layout changes (viewports ≥ 768px / `md:` and above) — the existing layout is correct.
- Performance optimization of the cipher background animation.
- Changes to app logic, state management, or non-visual behavior.
- New mobile-only features (gestures, swipe navigation, etc.).
- PWA manifest or app icon changes for mobile home screen installation.
- Accessibility improvements beyond tap target sizing in the header (US-3 AC-2).
- Support for viewports narrower than 360px.
